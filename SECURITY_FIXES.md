## 🟡 تحقق مدخلات صارم (OWASP API Security Top 10)
تمت إضافة `zod` مع مخططات تحقق صارمة لكل مسار حسّاس (تسجيل، دخول، OTP، محفظة، طلبات)، بحيث تُرفض أي حقول غير متوقعة (Mass Assignment) وتُفرض حدود منطقية (مثال: حد أقصى 50000 لكل عملية مالية). راجع `validation/schemas.js` و`middleware/validate.js`.

## 🔵 سجل تدقيق (Audit Log)
كل عملية إدارية حساسة (تعديل رصيد، حظر، رفع حظر، تفعيل/تعطيل حساب، تفعيل/تعطيل ميزة) تُسجَّل الآن في جدول `audit_logs` مع هوية الأدمن، الوقت، عنوان IP، والتفاصيل. متاحة عبر `GET /api/admin/audit-logs`.

## نظام OTP (رمز التحقق) عبر SMS وواتساب

### التصميم
- `models/Otp.js`: يخزّن **hash** الرمز فقط (bcrypt) وليس الرمز الصريح، مع صلاحية 5 دقائق وحد 5 محاولات تحقق خاطئة قبل الرفض.
- `services/otpService.js`: توليد الرمز بـ `crypto.randomInt` (آمن، غير قابل للتنبؤ)، مع فترة تهدئة 60 ثانية بين كل طلب رمز جديد لنفس الرقم.
- بوابتان جاهزتان للتوصيل الفعلي:
  - **SMS**: عبر `SMS_PROVIDER` في `.env` (يدعم أي مزوّد محلي ليبي أو Twilio بعد ربط الـ API الفعلي في `sendSms()`).
  - **واتساب**: عبر WhatsApp Business Cloud API (Meta) — `WHATSAPP_PROVIDER`, `WHATSAPP_PHONE_NUMBER_ID`, `WHATSAPP_API_TOKEN` في `sendWhatsapp()`.
- حماية من استنزاف رصيد البوابة: حد صارم 8 طلبات OTP لكل رقم/IP كل 10 دقائق (`routes/otpRoutes.js`).

### أين يُستخدم إلزاميًا الآن
1. **التسجيل**: `POST /api/auth/register` يرفض أي طلب بدون `otp_code` صالح لنفس رقم الهاتف (`purpose=register`).
2. **التحويل/الشحن الكبير**: أي عملية شحن أو تحويل فوق `OTP_REQUIRED_ABOVE_AMOUNT` (افتراضي 500 د.ل، قابل للتعديل) تتطلب رمز OTP إضافي (`purpose=wallet_transfer` / `wallet_topup`) حتى لو كان المستخدم مسجّل الدخول أصلًا — هذا "تحقق ثنائي" حقيقي على العمليات المالية الكبيرة تحديدًا، وليس على كل شيء، حفاظًا على تجربة استخدام سلسة.

### نقاط النهاية
| المسار | الوصف |
|---|---|
| `POST /api/otp/request` | `{ phone, purpose, channel }` — يرسل الرمز عبر SMS أو واتساب |
| `POST /api/otp/verify` | `{ phone, purpose, code }` — تحقق مستقل (تستخدمه واجهات الأدمن أو تدفقات أخرى) |

## ⚠️ ما لم يُنفَّذ بعد (يتطلب قرارًا منك)
- **ميزة "مراسلة خدمة العملاء"**: غير موجودة في التسليم الأصلي؛ سكربت اختبار الحمل (`scripts/load-test.js`) يحاكيها كمسار افتراضي فقط. أخبرني إن أردت بناءها الآن (تحتاج: نموذج تذاكر/محادثات + Socket.IO + على الأرجح طابور رسائل).
- **اختبار تحميل حقيقي بالأرقام المطلوبة (2500/1200/850)**: لا يمكن تنفيذه فعليًا في بيئة التطوير هذه (لا شبكة، لا قاعدة بيانات حيّة). تم تسليم سكربت k6 جاهز (`scripts/load-test.js`) بنفس الأرقام تمامًا لتشغيله على بيئة حقيقية (Staging).
- **Redis للكاش الموزّع ولـ Socket.IO adapter**: ضروري فقط عند تشغيل أكثر من نسخة سيرفر (PM2 cluster يشغّل عدة عمليات على نفس الجهاز وتعمل بشكل صحيح دون Redis لأن كل العمليات تتشارك نفس قاعدة البيانات، لكن الكاش الداخلي للـ feature flags سيكون منفصلًا لكل عملية — التأخير الأقصى للتزامن 30 ثانية وهو مقبول غالبًا؛ إن أردت تزامنًا فوريًا عبر عمليات متعددة أخبرني لأضيف Redis Pub/Sub).
